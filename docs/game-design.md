# TypeRush 游戏设计文档

> **说明**: 本文档定义《TypeRush》的核心玩法、游戏模式、功能清单与交互流程。面向开发团队，作为实现阶段的统一参考。

---

## 一、核心游戏机制

### 1.1 基本流程

```
选择模式 → 选择文本/参数 → 倒计时(3..2.1) → 开始输入 → 结束 → 结算页
```

每个游戏回合经历以上固定流程，单人与多人的区别主要在"结束条件"和"结算页"的呈现方式。

### 1.2 核心数据指标

| 指标 | 全称 | 计算公式 | 说明 |
|------|------|----------|------|
| WPM | Words Per Minute | (正确字符数 / 5) / (用时秒数 / 60) | 行业标准速度单位。1 word = 5 字符 |
| CPM | Characters Per Minute | 总输入字符数 / (用时秒数 / 60) | 按实际字符数计算的速度 |
| 准确率 | Accuracy | 正确字符数 / (正确字符 + 错误字符) × 100% | 百分比，不含退格 |
| 原始 WPM | Raw WPM | (总输入字符 / 5) / (用时秒数 / 60) | 不考虑退格的原始速度，反映真实输入速率 |
| 一致性 | Consistency | 各秒 WPM 的标准差 | 越低表示速度越稳定 |

### 1.3 输入处理规则

```
用户按键
    │
    ├── 普通字符 (a-z, 0-9, 标点)
    │      └── 与当前期待字符比较 → 正确 / 错误 / 多余
    │
    ├── 退格 (Backspace)
    │      └── 回到上一个未输入字符（不计入正确/错误统计）
    │
    ├── 空格 (Space)
    │      └── 跳过当前单词进入下一个
    │
    └── 特殊键 (Tab, Ctrl, Alt, 方向键)
           └── 忽略（无响应）
```

- 错误输入的字符以红色高亮标记
- 当前待输入的字符以特殊光标标记
- 已正确输入的字符显示为正常色
- 未输入的字符显示为灰色

### 1.4 文本显示与输入焦点

- 屏幕上显示 2-3 行文本（根据字号）
- 当前行位于屏幕中央，上下各保留 1 行供上下文参考
- 输入框获得焦点时即监听键盘事件，无需手动点击
- 游戏结束后输入焦点自动释放

## 二、单人模式

### 2.1 计时模式 (Time)

最经典的打字模式。设定一个时间限制，玩家在时间内尽可能多地输入。

| 参数 | 值 |
|------|----|
| 可选时长 | 15s / 30s / 60s / 120s |
| 文本来源 | 随机单词流（从词库中抽取，无限生成）|
| 结束条件 | 倒计时归零 |
| 特殊规则 | 最后一个单词：若倒计时结束正好在输入中，已完成的部分计入成绩 |
| 排行榜维度 | WPM、准确率 |

### 2.2 文字模式 (Words)

输入固定数量的单词，挑战在最少时间内完成。

| 参数 | 值 |
|------|----|
| 可选词数 | 10 / 25 / 50 / 100 / 200 |
| 文本来源 | 从词库随机抽取指定数量的单词 |
| 结束条件 | 完成指定词数 |
| 特殊规则 | 无 |
| 排行榜维度 | 用时、WPM、准确率 |

### 2.3 引文模式 (Quote)

输入一段名人名言或完整的句子/段落，测试在真实语境下的表现。

| 参数 | 值 |
|------|----|
| 可选长度 | short / medium / long |
| 文本来源 | 内置引文库（按长度、难度分类）|
| 结束条件 | 完整输入整段引文 |
| 特殊规则 | 包含标点符号和大写字母，考验全键盘输入能力 |
| 排行榜维度 | WPM、准确率、用时 |

### 2.4 代码模式 (Code)

输入一段代码片段，专为程序员设计。

| 参数 | 值 |
|------|----|
| 可选语言 | JavaScript / Python / Rust / Go / HTML / CSS / SQL / TypeScript |
| 文本来源 | 内置代码片段库 |
| 结束条件 | 完整输入整段代码 |
| 特殊规则 | 保留缩进、特殊字符（括号、分号、引号），不保留注释 |
| 排行榜维度 | WPM、准确率、用时 |

### 2.5 自定义模式 (Custom)

使用玩家自己输入的文本进行练习。

| 参数 | 值 |
|------|----|
| 文本来源 | 玩家手动输入或粘贴 |
| 结束条件 | 完整输入全文 |
| 特殊规则 | 文本持久化到用户账户，可重复使用 |
| 排行榜维度 | 不参与排行榜（仅供个人记录）|

### 2.6 专注模式 (Zen)

没有时间和目标限制，纯粹为了放松。

| 参数 | 值 |
|------|----|
| 时长 | 无限制 |
| 文本来源 | 随机单词流 |
| 结束条件 | 玩家主动停止（按 Esc 或点击结束）|
| 特殊规则 | 不显示 WPM 计时（可选设置），不记录成绩 |
| 排行榜维度 | 不参与排行榜 |

### 2.7 每日挑战 (Daily)

每天同一段文本，所有玩家挑战一次，按 WPM 排名。

| 参数 | 值 |
|------|----|
| 时长 | 统一使用 60s 计时模式 |
| 文本来源 | 每日自动生成或精选，所有玩家相同 |
| 参与限制 | 每人每天一次机会 |
| 排行周期 | 每日 0 点 UTC 重置 |
| 排行榜维度 | WPM（同分按准确率排序）|

---

### 2.8 单词学习模式 (Vocabulary)

将背单词与打字练习结合的特色模式。一边练打字速度，一边通过"拼写输出"加深单词记忆，实现练习与学习的双重目标。

| 参数 | 值 |
|------|----|
| 可选词库 | CET-4 / CET-6 / GRE / TOEFL / IELTS / 考研英语 / 高考 / 自定义词库 |
| 学习方式 | 显示单词 + 中文释义，玩家输入该单词完成拼写 |
| 输入模式 | **拼写模式**: 逐词出现，完整拼出该单词后进入下一个 |
| 　　　　 | **打字模式**: 如同计时模式，但文本来源限定为当前词库，边打边记 |
| 结束条件 | 完成本轮所有单词 / 设定时间结束 |
| 特殊规则 | 输入错误的单词自动加入临时复习列表，在同一次游戏中循环出现直到正确 |
| 复习模式 | 独立入口，只复习历史错误单词，采用间隔重复策略（类似 Anki / SuperMemo） |
| 排行榜维度 | 不参与公开排行（仅供个人记录）|
| 数据追踪 | 每个单词的错误次数、平均反应时间、掌握度 (%)、遗忘曲线 |
| 词库格式 | 支持 CSV / JSON 导入，格式: `单词, 释义, 例句（可选）` |

交互流程:

```
选择词库 → 选择模式（拼写/打字）→ 开始游戏
                            │
                   ┌────────▼────────┐
                   │   拼写模式       │
                   │                 │
                   │ 单词: ambiguous │
                   │ 释义: 模棱两可的 │
                   │ 输入: a_m_b_    │  ← 逐字母打字，实时比对
                   │ 进度: 3/20 词   │
                   └────────┬────────┘
                            │ 单词正确 → 进入下一个
                            │ 单词错误 → 加入复习队列
                            ▼
                    ┌──────────────┐
                    │   打字模式     │
                    │               │
                    │ 同计时模式，    │
                    │ 但文本来源     │
                    │ 限定为词库单词  │
                    └──────────────┘
```
## 三、多人模式

### 3.1 快速竞速 (Race)

最核心的多人模式。所有玩家输入同一段文本，先完成者胜。

| 参数 | 值 |
|------|----|
| 玩家人数 | 2-8 人 |
| 文本来源 | 系统随机选取引文/单词集 |
| 游戏时长 | 直到最后一名完成或超时（5min）|
| 胜负判定 | 1. 完成全部文本的用时最短 2. 若同时完成，准确率高者胜 |
| 特殊规则 | 玩家完成后可看到其他人的实时进度 |
| 匹配方式 | 快速匹配 或 创建私人房间 |

交互流程:

```
玩家A创建房间
    │
    ├── 快速匹配
    │      ↓
    │   匹配系统找到同段位玩家
    │      ↓
    │   集合到游戏房间
    │
    └── 私人房间
           ↓
        生成房间码 (6位)
           ↓
        玩家分享房间码
           ↓
        其他玩家输入房间码加入
           ↓
        房主点击"开始游戏"
```

### 3.2 限时竞速 (Time Battle)

在限定时间内比赛谁打得更多。

| 参数 | 值 |
|------|----|
| 玩家人数 | 2-8 人 |
| 文本来源 | 随机单词流（所有人相同）|
| 游戏时长 | 30s / 60s / 120s（房主设定）|
| 胜负判定 | 时间结束后，WPM 最高者胜 |
| 特殊规则 | 实时显示所有人的 WPM 排行榜，结束时按最终 WPM 排序 |

### 3.3 准确率挑战 (Accuracy Challenge)

比赛谁的准确率更高，速度次之。

| 参数 | 值 |
|------|----|
| 玩家人数 | 2-8 人 |
| 文本来源 | 短引文（50-100 字符）|
| 游戏时长 | 直到最后一名完成 |
| 胜负判定 | 1. 准确率最高者胜 2. 同分则用时短的胜 |
| 特殊规则 | 强制慢速——退格不计入修正，每个错误字符扣分 |

### 3.4 淘汰赛 (Elimination)

每隔一段时间淘汰当前速度最慢的玩家，直到剩下最后一人。

| 参数 | 值 |
|------|----|
| 玩家人数 | 4-8 人 |
| 文本来源 | 随机单词流 |
| 游戏时长 | 持续淘汰至剩 1 人 |
| 胜负判定 | 最后存活的玩家获胜 |
| 淘汰规则 | 每 30s 计算一次实时 WPM，最低者被淘汰；被淘汰玩家可以看到剩余玩家的进度 |

### 3.5 团队对战 (Team Battle)

红蓝两队比拼综合打字能力。

| 参数 | 值 |
|------|----|
| 玩家人数 | 4-8 人（每队 2-4 人）|
| 文本来源 | 随机单词流（两队相同）|
| 游戏时长 | 60s |
| 胜负判定 | 队伍平均 WPM + 准确率加权分，高者胜 |
| 特殊规则 | 队员之间不可见彼此的输入内容（防止抄答案），但可见队伍总进度 |

### 3.6 文字马拉松 (Marathon)

长时间持续输入，中途可退出，最终按总输入量排名。

| 参数 | 值 |
|------|----|
| 玩家人数 | 2-∞（无上限）|
| 文本来源 | 随机单词流 |
| 游戏时长 | 持续开放 24 小时 |
| 胜负判定 | 总输入字符数（或总 WPM 平均）|
| 特殊规则 | 玩家可随时加入/退出，成绩持续累积；每 5 分钟自动保存进度 |

### 3.7 警察抓小偷 (Cop & Robber Chase)

> 灵感源自金山打字通同名经典模式。两名玩家在一条纵向地图上追逐，打字速度直接决定移动距离。

**角色设定:**
- **警察 (追赶者)**: 从起点出发，需要比对方打得更快才能缩短距离
- **小偷 (逃跑者)**: 起点领先警察一段距离（约 20% 进度），需要保持打字速度优势逃到终点

| 参数 | 值 |
|------|----|
| 玩家人数 | **2 人（强制 1v1）** |
| 文本来源 | 同一段文本，两人同步输入同一篇引文或单词集 |
| 地图长度 | 相当于 50-150 个单词的距离 |
| 胜负判定 | 警察追上小偷（重叠位置）→ 警察胜；小偷到达终点线 → 小偷胜 |
| 游戏时长 | 无上限（取决于双方打字速度）|
| 移动规则 | 每正确输入 1 个单词，角色前进 1 格；每错误 1 个字符，后退 0.3 格 |
| 匹配方式 | 快速匹配 或 邀请好友 1v1 |

**特殊机制:**

```
警察追赶过程中距离越近，双方心理压力越大:
  · 距离 < 5 格: 屏幕边缘出现红色警告边框（双方可见）
  · 小偷每错误一次: 警察获得一个"冲刺"机会（下一次正确输入移动 x 1.5）
  · 警察每错误一次: 小偷获得一个"烟雾弹"（自身加速 2 秒）
```

**道具系统（进阶玩法，Phase 2 引入）:**

文本中随机插入特殊标记 `[ITEM]`，正确输入该标记的角色拾取道具。

| 道具 | 图标 | 效果 | 持续时间 |
|------|------|------|----------|
| 加速 | 火箭 | 自身移动速度 x 1.5 | 5 秒 |
| 减速陷阱 | 磁铁 | 对方移动速度 x 0.6 | 4 秒 |
| 护盾 | 盾牌 | 免疫一次对方施加的负面效果 | 触发即消 |
| 闪现 | 闪电 | 瞬间前移 8 格（约 5 个单词）| 即时 |

**地图主题（Phase 2 引入）:**

| 地图 | 视觉效果 | 特殊规则 |
|------|----------|----------|
| 城市巷道（默认）| 俯视街道夜景，霓虹灯光 | 无 |
| 森林小径 | 林间小道，月光洒落 | 每 20 格出现岔路（输入特定单词选路）|
| 屋顶追逐 | 城市天际线，楼宇之间跳跃 | 错误过多会"失足"（多后退 3 格）|
| 地下隧道 | 黑暗隧道，只有前方光亮 | 屏幕可视范围随进度逐渐缩小/扩大 |

**交互流程:**

```
匹配/邀请
    │
    ├── 快速匹配: 系统自动匹配同段位对手
    │      ↓
    └── 邀请好友: 选择在线好友发起对战
           ↓
    角色分配（警察/小偷/随机）
           ↓
    倒计时 3..2..1
           ↓
    ┌───────────────────────────────────────────┐
    │  警察  ████████████░░░░░░░░░░░░  42%      │
    │  小偷  ██████████████████░░░░░░  68%      │  ← 实时位置
    │  地图 ════════════════════●═══════════     │  ← 全览地图
    │                                           │
    │  ┌───────────────────────────────────┐    │
    │  │ the quick brown fox jumps over .. │    │  ← 共享文本
    │  │ lazy dog...                       │    │  ← 各自独立光标
    │  └───────────────────────────────────┘    │
    │                                           │
    │  WPM: 92 vs 85   准确率: 97% vs 95%       │  ← 实时对比
    └───────────────────────────────────────────┘
           │
           ▼
    结算页: 胜负展示 + 双方详细数据对比 + 重播回放
```
## 四、游戏房间系统

### 4.1 房间类型

| 类型 | 说明 |
|------|------|
| **快速匹配房间** | 系统自动创建，玩家通过匹配算法进入，不可设置参数 |
| **私人房间** | 玩家创建，可自定义所有参数，通过房间码邀请好友 |
| **公开房间** | 显示在房间列表中，任何人都可加入，房主可设定密码 |

### 4.2 房间状态机

```
               ┌──────────────┐
               │   Waiting    │  ← 等待玩家加入、房主设置参数
               └──────┬───────┘
                      │ 所有人准备就绪 / 房主点击开始
                      ▼
               ┌──────────────┐
               │  Countdown   │  ← 3..2..1 倒计时
               └──────┬───────┘
                      │
                      ▼
               ┌──────────────┐
               │   Playing    │  ← 游戏进行中
               └──────┬───────┘
                      │ 游戏结束条件触发
                      ▼
               ┌──────────────┐
               │  Result      │  ← 显示结果，可退出或再来一局
               └──────┬───────┘
               ┌───────┴────────┐
               │                 │
               ▼                 ▼
         ┌──────────┐    ┌──────────────┐
         │  Rematch  │    │  Disband     │  ← 回到大厅
         └──────────┘    └──────────────┘
```

### 4.3 房间设置参数

| 参数 | 可选值 | 默认值 | 说明 |
|------|--------|--------|------|
| 模式 | Race / Time Battle / Accuracy / Elimination / Team / Chase | Race | 游戏模式 |
| 文本来源 | 单词 / 引文 / 代码 | 单词 | 文本类型 |
| 文本难度 | 简单 / 普通 / 困难 | 普通 | 文本难度筛选 |
| 游戏时长 | 15s / 30s / 60s / 120s | 60s | 限时模式下生效 |
| 最大人数 | 2-8 | 8 | 房间最大容纳（Chase 模式固定为 2 人） |
| 密码 | 可选 | 无 | 私人房间密码 |
| AI 机器人 | 关闭 / 1-7 个 | 0 | 补充不足人数的 AI 对手 |

## 五、结算与统计

### 5.1 结果页面布局

每条游戏记录在结算页展示以下内容:

**个人信息区:**
- WPM（大字体，主视觉）
- 准确率
- CPM
- 用时
- 原始 WPM
- 一致性
- 字符统计卡片（正确 ✓ / 错误 ✗ / 遗漏 / 多余）

**多人对战额外信息:**
- 排名（带奖牌图标）
- 段位分变化（+15 / -8）
- 所有玩家的排名列表（WPM + 准确率）
- 进度回放（可查看每位玩家在时间线上的进度变化）

**历史对比区:**
- 本次成绩与个人历史平均对比（WPM 和准确率）
- 个人最好成绩标志

### 5.2 数据持久化

| 数据 | 存储位置 | 用途 |
|------|----------|------|
| 原始按键序列 | PostgreSQL | 历史回放、作弊审查 |
| 统计摘要 | PostgreSQL | 排行榜、个人主页 |
| 实时游戏状态 | Redis | 游戏进行中的房间状态 |
| 排行榜缓存 | Redis Sorted Set | 高效实时排行 |

## 六、用户系统

### 6.1 账户类型

| 类型 | 功能限制 | 说明 |
|------|----------|------|
| 游客 | 可游玩单人和快速匹配多人，不记录历史成绩 | 无需注册，自动分配临时用户名 |
| 注册用户 | 全部功能 | 保存游戏历史、排行榜、好友、自定义设置 |

### 6.2 个人主页

- 用户名、头像、注册时间
- 总游戏场数、总 WPM 平均、总准确率平均

## 八、UI/UX 设计要点

### 8.1 设计原则

- **极简**: 去除不必要的装饰元素，让玩家专注于文本和输入
- **沉浸**: 深色主题优先，减少视觉干扰
- **反馈即时**: 按键即时有视觉反馈（变色、光标移动、音效可配置）
- **信息层次清晰**: 最重要的信息（当前输入位置）最突出，统计数据次之

## 十一、参考设计借鉴 — jstris & tetr.io

> jstris (jstris.eu) 与 tetr.io 是当前最活跃的两款在线俄罗斯方块游戏。虽然品类不同（方块消除 vs 打字），它们在**多人匹配、房间系统、段位体系、UI/UX** 上的设计思路对打字游戏有很强的参考价值。

### 11.1 从 jstris 学到的设计

jstris 以极简、轻量、快速匹配著称。它的核心哲学是"最小化进入游戏的摩擦"。

| 可借鉴的设计 | 原始实现（jstris） | 应用到 TypeRush |
|-------------|-------------------|-----------------|
| **房间码系统** | 6 位字符房间码，创建即生成，分享给好友即可加入 | 私人房间使用同样机制，无需登录即可加入 |
| **公开房间大厅** | 显示所有活跃房间列表：房间名、模式、人数、状态 | 同思路——玩家可浏览所有公开房间，选择感兴趣的加入 |
| **快速匹配** | 点击 Quick Play 直接进入匹配队列，自动分配对手 | 同模式——点击即匹配，省去参数配置步骤 |
| **游客友好** | 不注册也能玩，部分功能受限但不影响核心体验 | 单人练习 + 快速匹配不需要登录，降低新用户门槛 |
| **极简游戏 UI** | 游戏中无任何多余元素，全部注意力在方块区域 | 打字时只显示待输入文本 + WPM + 进度条，其余全部隐去 |
| **回放系统** | 每局自动保存回放，可查看任何人（包括高手）的完整操作 | 每局保存按键序列回放，可慢速回放学习高手打字节奏 |
| **模式多样性** | sprint 40L / 20L / cheese / survival / ultra 等 | 单人模式的多种变体满足不同训练目标 |
| **自定义快捷键** | 完全自定义按键映射，DAS/ARR 微调 | 自定义光标行为、退格灵敏度、自动换行等打字偏好 |

jstris 最值得学习的不是某个具体功能，而是那种 **"进来就能玩，玩完就看数据"** 的流畅感。

### 11.2 从 tetr.io 学到的设计

tetr.io 代表了在线竞技游戏的"工业化"水平——系统的完善度和玩家留存力都更高。

| 可借鉴的设计 | 原始实现（tetr.io） | 应用到 TypeRush |
|-------------|-------------------|-----------------|
| **段位阶梯体系** | Z → D → C → B → A → S → SS → U → X，每个段位有 +/- 细分，视觉上带金属/钻石/彩虹等特效 | 打字段位从 打字新手 → 键盘侠 → 极速传说，每段位有专属徽章和颜色 |
| **观战模式** | 在大厅浏览所有正在进行的比赛，点击即可观战，实时看到双方操作 | 浏览正在进行的多人比赛，点击观战，实时看到每位玩家的打字进度和按键 |
| **BP 电池系统** | 排位赛需要消耗 BP，BP 随时间自然恢复；没 BP 就去打练习或休闲 | 段位赛消耗"精力值"，防止连续输掉无限掉分，鼓励休息后再战 |
| **装饰系统** | 皮肤、方块效果、背景音乐包、面板样式，全部可购买/解锁 | 光标皮肤、打字特效（粒子/火花/水墨）、背景主题、按键音效包、结果页卡片样式 |
| **赛季制度** | 每个赛季约 1-2 个月，赛季末根据段位发放奖励，赛季重置部分段位分 | 月度赛季，赛季段位奖励（限定光标皮肤/头像框/徽章）|
| **详细统计页面** | 分模式的胜率、APM、时间分布、对手分析、能力雷达图 | 分模式的 WPM 趋势、准确率趋势、各键位错误热力图、成绩分布曲线 |
| **在线状态与私聊** | 好友列表显示在线/游戏中/观战中/离线，支持 whisper 私聊 | 好友在线状态 + 游戏内轻量聊天 |
| **自定义房间高级设置** | 游戏速度、起始高度、垃圾行速度等几十项参数可调 | 自定义房间可设置文本来源/难度/时长/允许使用退格/是否显示实时排名等 |

### 11.3 关键设计原则对比

```
                  jstris 风格                    tetr.io 风格
                 ───────────                   ───────────
  入门门槛         极低 (游客即玩)              低 (注册后快速上手)
  游戏内 UI        几乎为零 (极简)              适度 (信息面板+特效)
  段位系统         ELO 数值 (赤裸)              阶梯段位 (游戏化包装)
  留存驱动力       追求更高排名                  段位+赛季+装饰
  自定义程度       按键+灵敏度                   全方面 (含视觉/听觉)
  社区氛围         Discord + 第三方             内置好友+频道

  TypeRush 定位: 前端体验学 jstris 的干净利落，后端系统学 tetr.io 的完整生态
```

### 11.4 这些设计对打字游戏意味着什么

| 打字游戏的独特场景 | jstris/tetr.io 中的对应 | 具体设计建议 |
|-------------------|------------------------|-------------|
| 打字过程需要全神贯注 | 游戏中零干扰 UI | 游戏进行中完全隐藏导航、社交提示、排行榜；按 Esc 才显示暂停菜单 |
| 多人对战中谁领先很关键 | 实时排名实时更新 | 借鉴 tetr.io 的实时排名列表，动态排序，排名变化用动画过渡 |
| 不同文本难度影响成绩 | 不同模式区分体验 | 文本难度分级展示（类似 tetr.io 的房间设置面板），玩家可预判 |
| 键盘映射是硬需求 | 自定义按键绑定 | 左右手习惯、退格/回车行为可配置，适配不同键盘布局 (QWERTY/Dvorak/Colemak) |
| 速度提升过程是正反馈核心 | 段位提升的仪式感 | 每升一段位，播放特效动画 + 成就弹窗（类似 tetr.io 的段位升级动画）|

---

> **本篇参考**: jstris 的房间系统与极简 UI 适合 MVP 阶段直接模仿；tetr.io 的段位、装饰、赛季系统适合 Phase 2/3 逐步引入。
